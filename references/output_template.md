# Total-Insight-For-PM 输出模板

适用目标：输出正式书面化的《产品判断书》或《方向判断书》，用于立项、优先级、范围收敛或策略定义。

## 标题要求

- 文档标题统一采用：`<命题名称>_机会洞察`
- 一级、二级、三级、四级章节之间都加入一条 Markdown 分割线：`---`
- 二级章节标题可直接写结论，格式优先采用：`<章节名>：<一句话结论>`

## 贯通规则

- `看行业`、`看用户`、`看竞争`、`看自己`共同回答机会是否存在、谁有需求、现有方案留下什么缺口、我们是否有资格承担。
- `看机会`只做收敛，必须从前四看推出主机会点，不新增无证据的新主题。
- `定控制点`、`定目标`、`定策略`都必须围绕同一个主机会点展开。
- 若`看机会`无法用一句话写清主机会点，停止输出正式文档，进入补证或失败报告。
- 若`3定`任一章节无法说明服务哪个机会点，判定为结构失焦，回到`看机会`重收敛。

## 强制要求

- 每一节必须输出结论句，不得只有材料堆叠。
- `看机会`必须收敛，不得把所有可能方向都保留为“可做”。
- `看机会`必须由`看行业`、`看用户`、`看竞争`、`看自己`推出。
- `3定`必须直接服务`看机会`中的主机会点。
- 输出必须严格停留在既定框架内，不得新增框架外章节。
- 正文只写已证实内容；推测、假设、证据缺口与后续补证方向统一进入`备注项`。
- 创新判断必须有证据支撑，不得编造。
- 不得使用“不是A，而是B”“不取决于A，而取决于B”“真实价值点不在A，而在B”“优先B然后A”“先做B再做A”这类对仗式判断句。
- 用户旅程图必须先给旅程表，再给 Mermaid 可视化图，不得只输出孤立 Mermaid 代码块。
- Mermaid 旅程图默认使用 `flowchart LR`，阶段从左到右排列；每个阶段都要完整表达发生 / 对应核心场景 / 痛点 / 期望四层信息。

## Mermaid 旅程图输出规则

- 图标题统一使用：`用户体验旅程`
- 阶段数量优先控制在 `5-8` 个，避免过碎
- 每个阶段固定采用以下信息顺序：
  - `阶段编号 + 阶段名称`
  - `发生：...`
  - `对应核心场景：...`
  - `痛点：...`
  - `期望：...`
- 阶段之间只表达主链路，不在主图中扩展支线分叉
- 每个字段使用短句，不写长段落
- Mermaid 图应服务于可视化扫描，不能与上方旅程表内容冲突

## 反向验证规则

- 正式判断书、高优先级方向判断、对外汇报稿默认执行反向验证。
- 红军 subagent 只输出异议和风险，不改写正文。
- coordinator 负责裁决异议、修订最终稿并保持单一文风。
- 反向验证最多`3`轮。
- 第`1`轮检查同题性与收敛性。
- 第`2`轮检查证据性与判断强度。
- 第`3`轮检查表达纪律与可执行性。
- 第`3`轮后仍未达到发布条件时，不输出正式判断书。

## 失败报告模板

```md
# <命题名称>_机会洞察_失败报告

## 一、未达成发布条件的原因

## 二、当前卡住的层级

- 结构 / 证据 / 判断 / 表达

## 三、已解决问题

## 四、未解决问题

## 五、需要补充的输入或研究材料

## 六、是否需要重新定义命题
```
